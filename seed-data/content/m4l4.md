# Kubernetes for Architects: Deployments, Probes, HPA

You list Docker/K8s on your CV — this lesson makes sure you survive a deep-dive on it. Architect interviews test the *operational model*, not kubectl trivia.

## The mental model
Declarative desired state + reconciliation loops. You declare "3 replicas of booking-api v2.1"; controllers continuously converge reality toward it. Every Kubernetes question reduces to this.

## What architects must know cold
- **Deployment → ReplicaSet → Pods**; rolling updates with `maxSurge`/`maxUnavailable`; instant rollback because the old ReplicaSet still exists.
- **Probes** (the classic interview question):
  - *Liveness* fails → container restarted (use for deadlock detection, keep it dumb).
  - *Readiness* fails → pod removed from Service endpoints, no traffic (use for "DB connection not ready", warmups).
  - *Startup* → patience window for slow-booting apps before liveness applies.
  - Anti-pattern to name: liveness probe that checks the database — one DB blip restarts your entire fleet.
- **Requests vs limits:** requests drive scheduling; limits cap usage. Memory over limit = OOMKill; CPU over limit = throttling. .NET note: the runtime reads cgroup limits for GC/threadpool sizing — set them.
- **HPA:** scales replicas toward a metric target (CPU, memory, or custom like queue depth via KEDA — queue-depth scaling for your M3 workers is the perfect example to cite).
- **ConfigMaps/Secrets, Services (ClusterIP/LoadBalancer), Ingress** — one sentence each is enough.

## The senior answers
"How do you do zero-downtime deploys?" → rolling update + readiness probes + `terminationGracePeriod` with graceful shutdown handling in ASP.NET (`IHostApplicationLifetime`). "How does it scale?" → HPA on the right metric — CPU for APIs, queue depth for workers.
