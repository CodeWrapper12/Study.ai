# Kafka for .NET: Partitions, Consumer Groups, Rebalancing

RabbitMQ moves **commands**; Kafka stores **facts**. JDs list both — you need fluency in when and why.

## Core model
- A **topic** is split into **partitions**; each partition is an append-only, ordered log.
- Producers choose a partition via a **key** (`hash(key) % partitions`). Same key → same partition → **ordering guaranteed only per key**. Choose `OrderId` as key and each order's events are ordered; global ordering doesn't exist and you say so proudly.
- **Consumer groups:** partitions are divided among a group's consumers. Max parallelism = partition count (pick partition counts with headroom: 12 or 24, not 2).
- **Offsets:** consumers track position per partition; committing offsets = progress checkpoint. Replay = rewind offsets — this is why Kafka feeds analytics, search indexing, and audit from one stream.
- **Rebalancing:** consumers joining/leaving redistributes partitions; processing pauses briefly. Long-running message handlers + rebalances = duplicate processing → idempotency again.

## .NET specifics (Confluent.Kafka)
```csharp
var config = new ConsumerConfig
{
    BootstrapServers = "localhost:9092",
    GroupId = "booking-analytics",
    EnableAutoCommit = false,           // commit after successful processing
    AutoOffsetReset = AutoOffsetReset.Earliest
};
```
Manual commit after processing gives at-least-once. MassTransit also rides Kafka via its Rider API if you want one abstraction across both brokers.

## When to choose what (interview answer)
RabbitMQ: work distribution, RPC-ish commands, per-message routing. Kafka: event streams multiple teams consume, replay/audit requirements, high-throughput ingestion (the 50K location-writes/sec from M2's lab — that's a Kafka shape).

## Lab
`docker compose` a single-broker Kafka. Produce `BookingEvent`s keyed by OrderId to a 6-partition topic; run two consumers in one group and watch partition assignment; kill one mid-stream and observe the rebalance in logs.
