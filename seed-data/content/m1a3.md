## 1. Sliding window: the "contiguous" keyword

Trigger words: *substring, subarray, contiguous, consecutive, window of size k, longest/shortest run satisfying X*. The insight: a brute force checks O(n²) windows, but adjacent windows share almost all their content — so **maintain the window's state incrementally** instead of recomputing it.

### Fixed-size window: the warm-up
```python
def max_window_sum(nums: list[int], k: int) -> int:
    window = sum(nums[:k])
    best = window
    for right in range(k, len(nums)):
        window += nums[right] - nums[right - k]    # add entering, remove leaving
        best = max(best, window)
    return best
```
O(n) instead of O(nk). The state here is a sum; in harder problems it's a Counter or a set — the *shape of the update* never changes.

### Variable-size window: derive it from the invariant
The template everyone memorizes — but you should be able to **re-derive**:

1. Define the window invariant (e.g., "no repeated characters").
2. Expand `right` greedily — always.
3. When the invariant breaks, shrink from `left` until it holds again.
4. Record the answer whenever the window is valid.

```python
def longest_unique_substring(s: str) -> int:
    seen, left, best = set(), 0, 0
    for right, ch in enumerate(s):
        while ch in seen:                          # invariant broken
            seen.remove(s[left]); left += 1        # shrink until restored
        seen.add(ch)
        best = max(best, right - left + 1)
    return best
```

**Why it's O(n) despite nested loops:** `left` only moves right, at most n times total across the entire run. Each pointer advances ≤ n times → ≤ 2n work. This is the same amortized argument as Longest Consecutive Sequence — notice the family resemblance and *say it* in interviews; pattern-connecting is exactly what distinguishes senior answers.

### The senior-tier variant: Longest Repeating Character Replacement
Window of letters where you may replace at most k characters. Invariant: `window_size − count_of_most_frequent ≤ k`.
```python
def character_replacement(s: str, k: int) -> int:
    counts = Counter()
    left = best = max_freq = 0
    for right, ch in enumerate(s):
        counts[ch] += 1
        max_freq = max(max_freq, counts[ch])
        if (right - left + 1) - max_freq > k:      # invalid: too many replacements needed
            counts[s[left]] -= 1
            left += 1                               # note: shrink by ONE, not a while-loop
    return best if (best := max(best, 0)) else (len(s) and (right - left + 1))
```
Wait — that last line is too clever to write under pressure. The honest version:
```python
def character_replacement(s: str, k: int) -> int:
    counts = Counter()
    left = max_freq = best = 0
    for right, ch in enumerate(s):
        counts[ch] += 1
        max_freq = max(max_freq, counts[ch])
        while (right - left + 1) - max_freq > k:
            counts[s[left]] -= 1
            left += 1
        best = max(best, right - left + 1)
    return best
```
**Lesson inside the lesson:** when you catch yourself writing something "clever," delete it. Interview code is judged on clarity-under-verification, not golf. (The subtle point worth knowing here: `max_freq` may go stale after shrinking, and the algorithm is *still correct* because a stale max only makes the window conservatively smaller — never reports a false best. If an interviewer probes that, walking through why staleness is safe is a genuinely impressive two minutes.)

### Minimum Window Substring — the boss fight
Two Counters (need / have) + a `formed` counter so validity checks are O(1), expand-then-contract. Attempt it only after the above are clean; it's the same template with heavier state bookkeeping, and it teaches you that **window state design is the whole game**.

## 2. Stacks: nesting and "most recent first"

Trigger words: *matching pairs, valid nesting, undo, most recent, next greater/smaller, evaluate expression*. A stack is the right tool whenever the **most recently seen unresolved thing** is the first thing you'll need to resolve.

### Valid Parentheses — and what it's actually testing
```python
def is_valid(s: str) -> bool:
    pairs = {')': '(', ']': '[', '}': '{'}
    stack = []
    for ch in s:
        if ch in pairs:
            if not stack or stack.pop() != pairs[ch]:
                return False
        else:
            stack.append(ch)
    return not stack            # leftover openers = invalid
```
The two bugs interviewers wait for: popping an empty stack, and forgetting the final emptiness check. Edge-case discipline *is* the assessment here.

### Min Stack — augment, don't search
O(1) `get_min` by pushing pairs `(value, min_so_far)` — or a parallel min-stack. The transferable principle: **carry derived state with the data instead of recomputing it** — it's the same instinct as a covering index or a denormalized read model. Connect those dots aloud if asked; it's a one-sentence bridge between this round and your day job.

## 3. Monotonic stacks: the senior separator

**Problem shape:** for each element, find the next/previous greater/smaller element. Brute force O(n²); monotonic stack O(n).

**The idea:** maintain a stack whose elements are always in monotonic order. Each new element *pops everything it dominates* — and the moment of popping is exactly the moment you learn the answer for the popped element.

### Daily Temperatures, fully narrated
```python
def daily_temperatures(temps: list[int]) -> list[int]:
    res = [0] * len(temps)
    stack = []                                  # indices; temps[stack] strictly decreasing
    for i, t in enumerate(temps):
        while stack and temps[stack[-1]] < t:   # today answers their question
            j = stack.pop()
            res[j] = i - j
        stack.append(i)
    return res
```
**The amortized argument, ready-to-say:** "Each index is pushed once and popped at most once, so despite the nested while, total operations are ≤ 2n → O(n)." Deliver that sentence smoothly and you've separated yourself from most candidates at the door.

**Why store indices, not values:** the answer needs *distance* (i − j), and indices recover values (`temps[j]`) but values can't recover positions. Default to indices in every monotonic-stack problem.

### Largest Rectangle in Histogram — where it leads
The hard endpoint of this family: for each bar, the widest rectangle it limits spans from its previous-smaller to its next-smaller bar — both delivered by one monotonic pass with a sentinel bar of height 0 appended to flush the stack. Don't grind it until Daily Temperatures and Next Greater Element are reflexive; then it's 20 focused minutes instead of a demoralizing hour.

## 4. How this breaks in production

Sliding windows are rate limiters: "max N requests in any 60-second window" is literally this lesson — and the naive fixed-bucket implementation has the boundary-burst flaw that a true sliding window (or sliding log) fixes. Monotonic structures power "max in sliding window" metrics in monitoring systems. When an interviewer asks "where would you use this?", you have real answers: the per-partner rate limiting on your partner API is a sliding-window counter.

## 5. Interview lens

- Window problems: **name the invariant first**, before any code. "The window is valid while …" — interviewers relax visibly when they hear it, because it means the while-loop will write itself.
- Stack problems: state what the stack *contains* and its ordering invariant as a comment on the line where you declare it. `# indices, temps strictly decreasing` prevents the mid-solution confusion that sinks candidates.
- Asked for the window *contents* and not just the size? You need the Counter/dict state version, not the set version — recognize which variant before coding.
