# RAG Done Properly: Chunking, Hybrid Search, Reranking

You've shipped LLM features (Semantic Kernel assistant, NL→SQL router). RAG is the next interview topic on every "AI-enabled architect" JD — and most candidates only know the demo version. This lesson is the production version.

## The uncomfortable truth
When RAG answers badly, the model is rarely the problem — **retrieval is**. Debug order: did the right chunk exist? was it retrieved? was it ranked high enough? did the prompt use it? Only then blame the LLM.

## Chunking
- Chunk along **semantic structure** (headings, sections, paragraphs), not blind character counts. A 512-token chunk that splits a table from its caption is self-sabotage.
- Overlap (10–20%) protects boundary context.
- Attach metadata (source, section, date, access level) — you'll filter on it, and access-level filtering is the difference between a demo and something legal lets you ship.

## Retrieval beyond naive vector search
- **Hybrid search:** BM25 keyword + vector similarity, fused (RRF). Keyword catches exact codes/IDs ("PNR ABC123") that embeddings blur; vectors catch paraphrase. Elasticsearch and pgvector+PostgreSQL FTS both do hybrid — note the synergy with your M4 Elastic skills.
- **Reranking:** retrieve top-50 cheaply, rerank to top-5 with a cross-encoder. Biggest single quality jump per engineering hour.
- **Query transformation:** rewrite the user's question before retrieval (expansion, HyDE). Your two-step router already does the cousin of this.

## Grounding & citations
Force the model to cite chunk IDs; render citations in the UI. Uncited claims get regenerated or flagged. This single pattern converts "the AI said something wrong" incidents into "click the source" conversations.
