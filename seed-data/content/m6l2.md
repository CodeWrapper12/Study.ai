# Agents & Function-Calling Patterns (Beyond Your 9 Plugins)

You already run 9 Semantic Kernel plugins in production — you're past the tutorial stage. This lesson is the architecture layer: what separates tool-calling from *agents*, and how to keep them safe.

## The spectrum (use this framing in interviews)
1. **Single-shot function calling** — model picks one tool, returns. (Your assistant today.)
2. **Router/workflow** — code controls the flow; LLM makes classified decisions at nodes. (Your two-step NL→SQL router.) Deterministic, testable, cheap. **Default to this.**
3. **Agent loop** — model observes tool results and decides next actions toward a goal, iterating until done. Powerful, but token-hungry and failure modes multiply: loops, derailment, tool misuse.

The architect's answer: *"I use the least agency that solves the problem."* Workflows where the path is known; agents only where genuine dynamism exists (open-ended investigation, multi-step research).

## Production agent patterns
- **Constrain the blast radius:** read-only tools by default; mutating tools require human approval or run in dry-run mode first. Scope credentials per tool, never god-mode.
- **Budgets:** max iterations, max tokens, wall-clock timeout. An agent without budgets is an unbounded cost and risk.
- **Structured tool results:** return typed errors the model can react to ("PERMISSION_DENIED" beats a stack trace).
- **State outside the loop:** persist plan/progress so a crashed agent resumes instead of restarting (the saga instinct from M3 — notice it transfers).
- **Observability:** log every tool call with inputs/outputs; trace the full loop (OpenTelemetry spans per step — your existing OTel habit extends directly).

## Lab
Take one real FastTrack assistant scenario that currently needs multiple user turns and redesign it (paper) as: (a) a fixed workflow, (b) a budgeted agent loop. Write the cost, latency, and failure-mode comparison — that comparison table is an interview artifact in itself.
