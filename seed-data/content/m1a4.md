# Binary Search Beyond Arrays & the Four Linked-List Mechanics

> **Learning objectives** — wield one binary-search template that never off-by-ones; recognize "binary search on the answer" problems (the senior-level variant); execute the four linked-list mechanics that compose into every list question.
>
> **Time:** ~70 min read + problem set.

---

## 1. Binary search is not about sorted arrays

The real definition: **if you can ask a yes/no question whose answer is monotonic over a search space — false, false, …, false, true, true, … — binary search finds the boundary in O(log n) questions.**

A sorted array is merely the most famous monotonic space. The pattern's actual reach: rotated arrays, answer spaces ("what's the minimum speed/capacity/time that works?"), even real systems — `git bisect` is binary search over commits where the question is "is the bug present?"; autoscaling capacity searches are the same shape.

### One template to end off-by-one errors forever

Pick **one** convention and never deviate. Mine: half-open `[lo, hi)`, converging to the boundary:

```python
def lower_bound(lo: int, hi: int, ok) -> int:
    """Smallest x in [lo, hi) where ok(x) is True; returns hi if none."""
    while lo < hi:
        mid = (lo + hi) // 2
        if ok(mid):
            hi = mid          # mid might be the answer; keep it in range
        else:
            lo = mid + 1      # mid is definitely not; exclude it
    return lo
```

Why this never breaks: the range strictly shrinks every iteration (`hi = mid` is strictly smaller because `mid < hi`; `lo = mid + 1` strictly grows), so no infinite loops; the invariant "answer ∈ [lo, hi)" holds throughout; loop ends when the range is empty and `lo` *is* the boundary. Classic find-x is `lower_bound` with `ok = lambda m: nums[m] >= x` plus an equality check. Every binary-search problem reduces to designing `ok`.

### Tier 2: Rotated sorted array — search on a broken invariant
`[4,5,6,7,0,1,2]`: not sorted, but **one half always is**. Identify the sorted half (compare `nums[lo]` vs `nums[mid]`), check whether the target lies inside it, recurse into the right half:

```python
def search_rotated(nums: list[int], target: int) -> int:
    lo, hi = 0, len(nums) - 1
    while lo <= hi:
        mid = (lo + hi) // 2
        if nums[mid] == target: return mid
        if nums[lo] <= nums[mid]:                       # left half sorted
            if nums[lo] <= target < nums[mid]: hi = mid - 1
            else: lo = mid + 1
        else:                                            # right half sorted
            if nums[mid] < target <= nums[hi]: lo = mid + 1
            else: hi = mid - 1
    return -1
```
(Closed-interval style here because we're hunting an exact value, not a boundary — knowing *when* each style fits is the actual skill.)

### Tier 3: Binary search **on the answer** — the senior tell
**Koko Eating Bananas:** minimum eating speed k to finish piles within h hours. The search space isn't the array — it's *k itself*, from 1 to max(piles). And `can_finish(k)` is monotonic: if speed k works, every faster speed works.

```python
import math

def min_eating_speed(piles: list[int], h: int) -> int:
    def ok(k: int) -> bool:
        return sum(math.ceil(p / k) for p in piles) <= h
    return lower_bound(1, max(piles) + 1, ok)
```

The recognition trigger: **"minimize the maximum" / "maximize the minimum" / "smallest X such that feasible(X)"**. Ship-packages-in-D-days, split-array-largest-sum, aggressive cows — all the same skeleton: monotonic feasibility + lower_bound. When you spot one in an interview, *say the pattern name*: "this is binary search on the answer — feasibility is monotonic in capacity." Naming it earns instant credibility.

## 2. Linked lists: four mechanics, every question

Linked lists test **pointer discipline under observation**, not cleverness. Four mechanics compose into everything:

### Mechanic 1: The three-pointer reversal dance
```python
def reverse_list(head):
    prev, cur = None, head
    while cur:
        nxt = cur.next        # save before breaking
        cur.next = prev       # reverse the arrow
        prev, cur = cur, nxt  # advance both
    return prev               # prev is the new head
```
Drill until your hands do it without your brain. It's a sub-step inside Reorder List, Palindrome List, Reverse-K-Group, and others — and fumbling it mid-composite-problem is how candidates drown.

### Mechanic 2: Slow/fast pointers
- **Middle of list:** fast moves 2, slow moves 1; when fast exits, slow is at the middle.
- **Cycle detection (Floyd):** if they ever meet, there's a cycle — O(1) space, the whole point versus a visited-set.
- **Cycle start:** after meeting, reset one pointer to head; advance both by 1; they meet at the cycle entrance. If asked why: from head to cycle-start is `a`, meeting point is `b` into the cycle, and the meeting condition gives `a ≡ (cycle_len − b)` — worth deriving once on paper so you can sketch it under pressure, not memorize it.
- **Kth from end:** lead one pointer by k, advance together.

### Mechanic 3: The dummy head
```python
dummy = ListNode(0, head)
# ... operate using dummy as anchor ...
return dummy.next
```
Eliminates every "what if I must modify the head?" special case — Remove-Nth-Node, Merge-Two-Lists, partitioning. Using it unprompted reads as experience; explaining *why* ("it makes the head a non-special node") reads as seniority.

### Mechanic 4: Merge two sorted lists
The zipper walk with a dummy head + tail pointer. Memorize it as a unit because **Merge K Sorted Lists = this mechanic + a heap of (value, list_index, node)** — a beautiful composition question that touches Lesson 6's heaps. (Python wrinkle: nodes aren't comparable, so the tuple needs a tiebreaker index — a detail that has sunk many otherwise-correct solutions.)

### Composite example: Reorder List (L0→Ln→L1→Ln-1…)
Find middle (Mechanic 2) → reverse second half (Mechanic 1) → zipper-merge the halves (Mechanic 4). Three rehearsed mechanics, zero new ideas. This is *the* template for how hard list questions decompose — and how you should narrate them: "I'll compose three standard steps: find middle, reverse back half, interleave."

## 3. How this breaks in production

Binary search on the answer is capacity planning: "minimum number of workers to drain this queue within SLA" is Koko with consumers instead of bananas — and you can say exactly that in interviews. `git bisect` over a regression is lower_bound with `ok = "bug present"`. Floyd's cycle detection has found infinite loops in linked structures from ORM lazy-loading graphs to workflow-engine state machines; cyclical references are not an academic concern.

## 4. Interview lens

- Announce your interval convention before coding: "I'll use half-open [lo, hi)." It pre-empts the off-by-one scrutiny entirely.
- For answer-space searches, define `ok()` as a named helper *first* — it shows you see the structure, and the main loop becomes three lines.
- For lists, draw boxes-and-arrows for anything beyond a single mechanic. Interviewers expect drawing; silent pointer manipulation is where bugs hide.
- Always voice the recursion-vs-iteration choice: "I'll reverse iteratively — O(1) space versus O(n) stack."
