# Scaling Reads: Caching, Replication & CDNs

Reads dominate most systems, so read-scaling comes up in nearly every design interview. Three layers, in order of reach:

## 1. Caching
- **Cache-aside** (what you've done with Redis/LazyCache): app checks cache → miss → DB → populate. Simple, but first-request latency and stampede risk.
- **Stampede protection:** jittered TTLs, per-key rebuild locks, or request coalescing. Mentioning this unprompted is a strong senior signal.
- **Invalidation:** TTL-based (eventual freshness) vs explicit invalidation on write (complex but precise). Say the famous line, then actually answer it.
- Know your eviction policies: LRU default, LFU for skewed popularity.

## 2. Read replicas
Primary takes writes, replicas serve reads. The cost: **replication lag** → a user may not see their own write. Fixes: read-your-own-writes routing (session pins to primary briefly), or monotonic reads via sticky replica. Tie this to your real experience running reporting reads against replicas.

## 3. CDN
For static and semi-static content; cache-control headers, signed URLs for private assets, edge TTLs. One sentence in an interview is usually enough unless the system is media-heavy.

## The senior move
Always state *what staleness the business can tolerate* before choosing. "Search results can be 30s stale, the booking state cannot" — that single sentence shows architect-level thinking.

## Lab
Sketch the read path for FastTrack Relay's partner-facing availability endpoint at 10× current load: what's cached, where, TTLs, and how a booking write invalidates it. One page.
