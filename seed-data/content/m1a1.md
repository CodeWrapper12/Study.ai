## 1. Why Python at all — the honest case

You write C# faster than 99% of engineers. So why switch for interviews?

Count the keystrokes. A dictionary-of-lists in C#:

```csharp
var groups = new Dictionary<string, List<string>>();
if (!groups.ContainsKey(key)) groups[key] = new List<string>();
groups[key].Add(value);
```

The same in Python:

```python
groups = defaultdict(list)
groups[key].append(value)
```

In a 35-minute coding round you'll write 40–80 lines. Python versions of typical solutions are **30–50% shorter**, and every saved line is thinking time. Interviewers at Microsoft, Amazon, and Careem are language-agnostic in scoring — but *you* are not language-agnostic in speed. There's a second-order effect too: most editorial solutions, mock partners, and prep material default to Python, so your study loop gets faster as well.

The counter-case, honestly stated: if a loop lets you use C# **and** the role is .NET-specific (most Gulf enterprise roles), use C#. The Python investment is for big-tech generalist loops where 20 minutes of boilerplate-free thinking wins offers. You're running both tracks, so you'll have both.

## 2. The translation layer — C# → Python, permanently

Don't learn Python "from scratch"; map it onto what you know. Internalize this table until lookups stop:

| You think (C#) | You write (Python) | Watch out |
|---|---|---|
| `Dictionary<K,V>` | `dict` / `defaultdict` | KeyError on missing key — use `.get(k, default)` or defaultdict |
| `HashSet<T>` | `set` | `{}` is an empty **dict**; empty set is `set()` |
| `List<T>` | `list` | negative indexing `xs[-1]` is your friend |
| `Queue<T>` | `collections.deque` | `popleft()` is O(1); `list.pop(0)` is **O(n)** — classic trap |
| `Stack<T>` | plain `list` | `append` / `pop()` from the end |
| `PriorityQueue<T,P>` | `heapq` on a list | **min**-heap only; push `(-priority, item)` for max |
| `StringBuilder` | `parts = []` … `''.join(parts)` | `s += piece` in a loop is O(n²) |
| `nums.OrderBy(x => x.Age)` | `sorted(nums, key=lambda x: x.age)` | `sorted` returns new list; `.sort()` mutates |
| `int.MaxValue` | `float('inf')` | works in comparisons; use `math.inf` equivalently |
| `(a, b) = (b, a)` swap | `a, b = b, a` | tuple unpacking everywhere — embrace it |
| `foreach (var (i, x) in xs.Index())` | `for i, x in enumerate(xs)` | also `zip(xs, ys)` for parallel iteration |
| ternary `c ? a : b` | `a if c else b` | order differs — read it aloud once |

Five idioms that make solutions read like editorial answers:

```python
# 1. Comprehensions (LINQ Select/Where in one breath)
squares = [x * x for x in nums if x % 2 == 0]

# 2. Counter — frequency maps for free
from collections import Counter
counts = Counter("engineering")          # {'e': 3, 'n': 3, ...}
counts.most_common(2)                    # top-k in one call

# 3. Unpacking with star
first, *rest = nums

# 4. Dict iteration
for key, value in groups.items(): ...

# 5. Custom sort by multiple keys (OrderBy/ThenByDescending)
intervals.sort(key=lambda iv: (iv[0], -iv[1]))
```

## 3. Complexity reality — what's actually O(1)

Saying "dict lookup is O(1)" is table stakes. Knowing the *fine print* is the senior signal:

- `dict` / `set`: average O(1) insert/lookup/delete — **amortized**, with O(n) worst case under adversarial hashing. In an interview say "average O(1)" and move on; if probed, mention open addressing and rehashing on growth.
- `list`: index O(1), `append` amortized O(1), **`insert(0, x)` and `pop(0)` are O(n)** — the single most common complexity lie in candidate code. Need both ends? `deque`.
- `in` operator: O(1) on set/dict, **O(n) on list** — `if x in my_list` inside a loop silently makes you O(n²).
- Slicing: `xs[a:b]` **copies** — O(b−a) time *and* memory. Recursing with `s[1:]` turns O(n) into O(n²); pass indices instead.
- `sorted` / `.sort()`: Timsort, O(n log n) worst, O(n) on nearly-sorted input, stable. "Stable" matters: equal keys keep relative order, which multi-key sort tricks rely on.
- String concat in a loop: O(n²). `''.join(parts)`: O(n). Non-negotiable.
- `heapq`: push/pop O(log n), `heapify` O(n) — building a heap from a list is linear, a fact that changes several top-k complexity answers.
- Recursion depth: default limit ~1000. A DFS on a 10⁶-node graph **will** crash; know the iterative rewrite (explicit stack) cold. Mentioning this unprompted is a strong systems-thinking signal.

## 4. The grind system — engineering your own training

You have a job, a side platform, and a family timezone. The system below is built for exactly that constraint, and it's the same spaced-repetition machinery you'd design for any learning service — you're just the user this time.

**The protocol:**

1. **Daily 60–90 minutes, non-negotiable, mornings if possible.** One focused problem beats three rushed ones. Consistency builds the pattern-recognition cortex; weekend marathons build burnout.
2. **Pattern-ordered, not random.** Follow this module's sequence (arrays/hashing → two pointers → window/stack → search/lists → trees/graphs → heaps/backtracking/DP). Random problem selection is the #1 self-study failure: you never get enough reps on one pattern for it to stick.
3. **The 25-minute rule.** Stuck for 25 minutes → read the *approach* (not the code) → close it → implement from a blank editor. Reading solutions feels like progress; reconstructing them *is* progress.
4. **Spaced repetition on failures.** Keep a failure log. Re-solve every failed problem from scratch at **+3 days, +7 days, +21 days**. A problem graduates only after a clean 21-day solve. This single habit separates people who've "done 300 problems" from people who can solve 30 under pressure.
5. **Speak aloud from week one.** Narrate every practice solve as if an interviewer were present. The skill being tested is *thinking aloud while coding*, and it is trainable and separate from problem-solving. Silent practice trains the wrong skill.
6. **Track ruthlessly.** Sheet columns: date, problem, pattern, result (clean / hints / failed), minutes, next-review date. Twenty seconds of logging per problem powers the whole system.

**Volume calibration for your 15 weeks:** ~120–150 problems total — 60% medium, 25% easy (speed reps), 15% hard (pattern extensions, not contest puzzles). Senior loops are decided on *flawless mediums with clear communication*, not on hards.

## 5. The five traps that eat C# developers

1. **Mutable default arguments.** `def f(x, acc=[])` — that list is created once and shared across calls. Use `acc=None` + `acc = acc or []`. This appears in backtracking templates constantly.
2. **Late-binding closures.** `funcs = [lambda: i for i in range(3)]` — all return 2. Bind with default arg: `lambda i=i: i`.
3. **Integer division.** `7 / 2 == 3.5` (float!), `7 // 2 == 3`, and `-7 // 2 == -4` (floors toward −∞, unlike C#'s truncation toward 0). Binary search mids: always `//`.
4. **No `++`.** `i += 1`. Your fingers will betray you for two weeks; let them.
5. **Chained comparison surprise.** `0 < x < 10` works and is idiomatic — but `a == b == c` means `a==b and b==c`, occasionally not what you meant when pasting logic.

## 6. Interview lens

How this lesson's material actually gets probed:

- *"What's the complexity of your solution?"* — answer for time **and** space, including the recursion stack. "O(n) time, O(h) space for the recursion where h is tree height" is the senior phrasing.
- *"Why a deque and not a list?"* — because `pop(0)` shifts every element, O(n) per op. One sentence, confident, move on.
- *"Could hashing degrade?"* — "Average O(1); worst case O(n) under pathological collisions; CPython mitigates with randomized string hashing." That's depth nobody expects — delivered in ten seconds.
