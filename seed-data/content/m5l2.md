# Terraform Fundamentals for Azure

Every senior JD now says "IaC (Terraform/Bicep)". Terraform is the cross-cloud lingua franca and the safer CV keyword. The concepts transfer to Bicep in an afternoon.

## Core loop
`write → terraform plan → review the diff → terraform apply`. The **plan is the product**: it's your change-review artifact, your PR body, your audit trail.

## The pieces
```hcl
terraform {
  required_providers { azurerm = { source = "hashicorp/azurerm", version = "~> 4.0" } }
  backend "azurerm" {                     # remote state with locking — non-negotiable in teams
    resource_group_name  = "rg-tfstate"
    storage_account_name = "stjsatfstate"
    container_name       = "tfstate"
    key                  = "relayflow.tfstate"
  }
}

resource "azurerm_resource_group" "main" {
  name     = "rg-relayflow-${var.environment}"
  location = "uaenorth"
}
```

- **State** maps declared resources to real ones. Remote backend + locking prevents two engineers applying simultaneously. State contains secrets → treat it as sensitive.
- **Variables + modules:** modules are your reusable components ("our standard AKS cluster"); environments (dev/prod) are the same modules with different `tfvars`. This is DRY for infrastructure — the same instinct you apply in C#.
- **Import & drift:** `terraform plan` exposes manual portal changes (drift); `import` adopts existing resources. Real organizations are brownfield — mentioning drift management is a senior signal.
- **Anti-patterns to name:** secrets in `.tfvars` committed to git (use Key Vault), one giant state file for everything (blast radius), portal "hotfixes" that drift silently.

## Lab
Terraform a resource group + PostgreSQL Flexible Server + Key Vault into your Azure free tier. Make one manual portal change, run `plan` to catch the drift, then revert it via `apply`. Destroy everything after — that's the IaC superpower, and the free-tier-friendly habit.
