# Polyglot Persistence: Mongo, Elasticsearch & Redis Patterns

Your gap analysis flagged this directly: JDs ask for Cosmos/Mongo/Elastic alongside SQL Server. The skill isn't knowing each database — it's **justifying the assignment of data to stores**.

## The decision framework (memorize this table)
| Data shape | Store | Why |
|---|---|---|
| Transactional aggregates, relations, reports | PostgreSQL/SQL Server | ACID, joins, your home turf |
| Self-contained documents, flexible schema, aggregate-per-read | MongoDB / Cosmos DB | Document = aggregate, horizontal scale built-in |
| Full-text search, relevance, faceting, fuzzy | Elasticsearch | Inverted indexes; SQL `LIKE '%x%'` does not scale |
| Hot keys, sessions, counters, geo, leaderboards | Redis | Microsecond reads, rich data structures |

## The architecture rule that keeps you safe
**One system of record; everything else is a derived, rebuildable projection.** Bookings live in PostgreSQL. The search index, the cache, the analytics store are all fed *from* it — via the events/outbox you built in M3 (or CDC like Debezium). Never dual-write from the app to two stores; that's the dual-write problem again wearing a different hat.

This sentence wins interviews: *"Elastic is a projection — if it's wrong or lost, I rebuild it from the source of truth, so its consistency requirements are eventual by design."*

## Cosmos DB notes (Azure-heavy JDs)
Partition key choice = the whole game (same logic as Kafka keys and shard keys — notice the pattern). RU/s pricing means a bad partition key is a billing incident, not just a perf one. Know the consistency spectrum: strong → bounded staleness → session (default, read-your-own-writes) → eventual.
