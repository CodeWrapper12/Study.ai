# gRPC in .NET: Contracts, Streaming, Deadlines

REST is your public face; gRPC is increasingly the default *between* services. JDs list it; interviews probe whether you know **when and why**, not just syntax.

## Why gRPC internally
- **Protobuf:** binary, ~5–10× smaller than JSON, schema-first contracts shared as a NuGet/proto package.
- **HTTP/2:** multiplexed streams over one connection — no head-of-line blocking at the HTTP level.
- **Generated clients:** no hand-written HttpClient wrappers, compile-time contract safety.
- **Four call types:** unary, server-streaming (live updates), client-streaming (bulk ingest), bidirectional.

## Contract evolution rules (the interview trap)
Field numbers are the wire identity. Never reuse or renumber a published field — `reserved 3;` retired ones. Adding fields is safe (unknown fields are ignored); renaming is safe (names aren't on the wire); changing types is not.

```proto
service BookingService {
  rpc GetBooking (GetBookingRequest) returns (BookingReply);
  rpc WatchStatus (GetBookingRequest) returns (stream StatusUpdate);
}
```

## Deadlines & cancellation — the senior detail
Every gRPC call should carry a deadline; it propagates down the call chain, so a 2s budget at the gateway becomes "whatever's left" at hop three. In .NET: `CancellationToken` flows automatically from `ServerCallContext`. Without deadlines, one slow dependency hangs the whole chain — say this in interviews and you sound like you've been burned (architects always have).

When NOT to use gRPC: public browser-facing APIs (grpc-web exists but REST/JSON wins for partners — your FastTrack partner API made the right call).
