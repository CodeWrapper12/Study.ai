# Arrays, Hashing & Two Pointers: The Decision Tree

> **Learning objectives** — recognize in under 30 seconds when a problem is a hash-map trade, a sort-then-scan, or a two-pointer convergence; implement the five canonical solutions from memory; articulate the space/time trade-off for each choice out loud.
>
> **Interview frequency:** ~⅓ of all screening questions are this family in costume. **Time:** ~75 min read + problem set.

---

## 1. The core insight: arrays are about *what you remember while scanning*

Every array problem is a single question: **as I walk this array, what minimal information must I remember to answer without walking it again?** The answer determines the pattern:

- Remember *which values I've seen* (existence) → **set**
- Remember *where I saw them* (position) → **dict value → index**
- Remember *how many times* (frequency) → **Counter**
- Remember *a running aggregate* (sum/min/max so far) → **one variable** (prefix idea)
- The array is sorted, so position itself encodes information → **two pointers / binary search**

Train yourself to ask this question explicitly. It converts "I don't know where to start" into a four-way multiple choice.

## 2. Hashing: buying O(1) memory of the past

### Canonical: Two Sum
```python
def two_sum(nums: list[int], target: int) -> list[int]:
    seen = {}                                  # value -> index
    for i, n in enumerate(nums):
        complement = target - n
        if complement in seen:                 # O(1) average
            return [seen[complement], i]
        seen[n] = i                            # store AFTER checking: handles duplicates
                                               # and "can't use same element twice"
    return []
```
Why store-after-check matters: for `nums=[3,3], target=6`, checking first finds the *earlier* 3 before overwriting. Tiny detail; interviewers watch for it.

### Canonical: Group Anagrams — the *signature key* idea
```python
def group_anagrams(strs: list[str]) -> list[list[str]]:
    groups = defaultdict(list)
    for s in strs:
        key = tuple(sorted(s))                 # canonical form as dict key
        groups[key].append(s)
    return list(groups.values())
```
The transferable concept is **canonicalization**: map every input to a normal form so that "equivalent" becomes "equal". Sorted-tuple key is O(k log k) per word; a 26-count tuple key is O(k) — mention both, code the simpler one, state the optimization. That sequencing (correct → state the better) is exactly what senior scoring rubrics reward.

### Canonical: Longest Consecutive Sequence — the *only-start-counting-at-heads* trick
```python
def longest_consecutive(nums: list[int]) -> int:
    s, best = set(nums), 0
    for n in s:
        if n - 1 not in s:                     # n is a sequence head
            length = 1
            while n + length in s:
                length += 1
            best = max(best, length)
    return best
```
This *looks* O(n²) — nested loops! — but each element is visited at most twice (once as a head-check, once inside exactly one sequence walk), so it's O(n). **Amortized analysis articulated aloud is a level-up moment in interviews.** Practice saying it: "the inner while only runs for sequence heads, and every element belongs to exactly one sequence, so total inner work is O(n)."

### Prefix sums — the aggregate variant
`Product of Array Except Self`, `Subarray Sum Equals K`: remember a running prefix and ask "have I seen `prefix − k` before?" — Two Sum wearing a trenchcoat:
```python
def subarray_sum(nums: list[int], k: int) -> int:
    seen = {0: 1}                              # prefix value -> count; 0 seen once (empty prefix)
    prefix = count = 0
    for n in nums:
        prefix += n
        count += seen.get(prefix - k, 0)
        seen[prefix] = seen.get(prefix, 0) + 1
    return count
```
The `{0: 1}` initialization handles subarrays starting at index 0 — the #1 bug in this problem.

## 3. Two pointers: when position carries information

Hashing trades **space for time**. Two pointers trades **a sort (or given sortedness) for space** — O(1) extra memory. Say this trade-off framing out loud; it's the architect's instinct applied to algorithms.

### Converging pointers (ends → middle): requires sorted or symmetric structure
```python
def two_sum_sorted(nums: list[int], target: int) -> list[int]:
    lo, hi = 0, len(nums) - 1
    while lo < hi:
        s = nums[lo] + nums[hi]
        if s == target: return [lo, hi]
        if s < target: lo += 1                 # need bigger -> move left pointer right
        else: hi -= 1                          # need smaller -> move right pointer left
    return []
```
The invariant — *every move provably discards only non-answers* — is what makes it correct. For **Container With Most Water**, the same logic: always move the **shorter** wall, because keeping it can never beat the current best (width only shrinks, height is capped by the shorter wall). Be ready to *prove* that in one sentence; it's the standard follow-up.

### 3Sum: pattern composition
Sort (O(n log n)) → fix `i` → two-pointer the remainder. Total O(n²). The hard part is **duplicate handling**:
```python
def three_sum(nums: list[int]) -> list[list[int]]:
    nums.sort()
    res = []
    for i in range(len(nums) - 2):
        if i > 0 and nums[i] == nums[i - 1]:   # skip duplicate anchors
            continue
        lo, hi = i + 1, len(nums) - 1
        while lo < hi:
            s = nums[i] + nums[lo] + nums[hi]
            if s < 0: lo += 1
            elif s > 0: hi -= 1
            else:
                res.append([nums[i], nums[lo], nums[hi]])
                while lo < hi and nums[lo] == nums[lo + 1]: lo += 1   # skip dup pairs
                while lo < hi and nums[hi] == nums[hi - 1]: hi -= 1
                lo += 1; hi -= 1
    return res
```

### Same-direction pointers (reader/writer): in-place compaction
`Remove Duplicates`, `Move Zeroes`: a slow "write" pointer and fast "read" pointer. This is the in-place partitioning idea underlying quicksort's partition step — connect them when asked.

## 4. How this breaks in production (yes, really)

This isn't just interview theater. The `if x in my_list` O(n²) trap from Lesson 1? You've seen its cousin in production: a report doing a correlated subquery per row — your own 60-second `vw_booking_transactions` fix was *exactly* a hash-join-vs-nested-loop decision made by an optimizer. When you explain hashing in interviews, you can land this: "it's the same reason I rewrote a correlated subquery to an OUTER APPLY with hash semantics — remembered lookups beat repeated scans." Cross-domain stories like that are senior-candidate gold.

## 5. Interview lens: the script for this family

1. Restate + constraints: sorted? duplicates? value ranges? negative numbers? size (does O(n²) pass)?
2. Brute force aloud with complexity — *always*, even when you see the optimal. It proves systematic thinking and buys time.
3. Name the trade: "I'll spend O(n) memory on a hash map to get one pass" or "it's sorted, so two pointers gives O(1) space."
4. Code with narration. Dry-run one normal case + one edge (empty, single element, all duplicates, negative numbers).
5. State final complexity unprompted.

Common follow-ups: "what if the array is sorted?" (switch to pointers, save memory) · "what if it can't fit in memory?" (external sort / streaming with bounded state — connect to your ETL instincts) · "what if values are bounded 0–100?" (counting array beats hash map).
