# Why Queues: RabbitMQ Fundamentals + MassTransit

This module closes your single biggest JD gap. Every Gulf Lead/Architect posting lists messaging; FastTrack Relay's domain events currently live in-process — after this module you can speak (and demo) the distributed version.

## RabbitMQ mental model
- Producers publish to **exchanges**; exchanges route to **queues** via bindings; consumers read queues.
- Exchange types: **direct** (exact routing key), **topic** (`orders.*.confirmed` wildcards), **fanout** (broadcast).
- Acknowledge after processing (`ack`); unacked messages are redelivered. Prefetch controls how many unacked messages a consumer holds — your concurrency throttle.

## MassTransit: the .NET way
Raw clients are plumbing; MassTransit gives you message contracts as C# records, automatic exchange/queue topology from types, retry policies, error queues, and (next lessons) outbox + sagas.

```csharp
public record BookingConfirmed(Guid OrderId, string PartnerCode, DateTime ConfirmedAt);

public class BookingConfirmedConsumer : IConsumer<BookingConfirmed>
{
    public async Task Consume(ConsumeContext<BookingConfirmed> ctx)
        => await _notifier.SendAsync(ctx.Message.OrderId);
}

services.AddMassTransit(x =>
{
    x.AddConsumer<BookingConfirmedConsumer>();
    x.UsingRabbitMq((ctx, cfg) =>
    {
        cfg.UseMessageRetry(r => r.Exponential(5, TimeSpan.FromSeconds(1),
            TimeSpan.FromMinutes(1), TimeSpan.FromSeconds(5)));
        cfg.ConfigureEndpoints(ctx);
    });
});
```

Failed-beyond-retries messages land in `*_error` queues automatically — that's your DLQ.

## Lab
`docker run rabbitmq:3-management`, then build a two-service solution: Publisher API emits `BookingConfirmed`; Worker consumes it. Kill the worker, publish 100 messages, restart — watch them drain. Throw on every 5th message and inspect the error queue in the management UI (port 15672).
