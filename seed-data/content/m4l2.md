# YARP API Gateway & the BFF Pattern

You built FastTrack's partner API *inside* the service. The next architectural rung is the **edge layer** — and YARP lets you build it in the stack you already master.

## What a gateway owns
Routing, TLS termination, authentication (validate JWT once at the edge), rate limiting (per partner!), request/response transforms, retries with circuit breaking, and centralized observability. Business logic stays OUT — the moment a gateway knows what a "booking" is, you've built a distributed monolith's choke point.

## YARP in 20 lines
```csharp
builder.Services.AddReverseProxy()
    .LoadFromConfig(builder.Configuration.GetSection("ReverseProxy"));
app.MapReverseProxy();
```
```json
"ReverseProxy": {
  "Routes": {
    "bookings": { "ClusterId": "booking", "Match": { "Path": "/api/bookings/{**catch-all}" } }
  },
  "Clusters": {
    "booking": { "Destinations": { "d1": { "Address": "http://booking-api:8080/" } },
                 "HealthCheck": { "Active": { "Enabled": true, "Path": "/health" } } }
  }
}
```
Because YARP is ASP.NET middleware, your existing auth, rate limiting (`AddRateLimiter`), and OpenTelemetry just plug in. That's its pitch versus nginx/Envoy: extensibility in C# rather than Lua/config DSLs. Trade-off to mention: you own its scaling and it's another .NET service to patch.

## BFF (Backend-for-Frontend)
One gateway per client *type*: the mobile BFF aggregates three service calls into one payload and handles cookie-based token storage; the partner BFF exposes the versioned, rate-limited, ACL-wrapped surface. Different clients evolve independently — this is the pattern name for what you already did instinctively with FastTrack's channel separation. Now you can name it in interviews.
