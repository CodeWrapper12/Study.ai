# The Outbox Pattern & Exactly-Once Myths

The interview question that filters architects: *"You save the order AND publish the event. The broker is down after the DB commit. Now what?"*

## The dual-write problem
DB commit and broker publish are two systems — no shared transaction. Any ordering can lose one side: state saved but event lost (downstream never reacts), or event published but state rolled back (downstream reacts to nothing).

## The outbox pattern
Write the event **into the same database transaction** as the state change, into an `outbox` table. A relay (background worker or MassTransit's built-in EF outbox) reads pending rows and publishes, marking them sent. The DB transaction is the single source of truth; publishing becomes at-least-once with retries.

```csharp
x.AddEntityFrameworkOutbox<AppDbContext>(o =>
{
    o.UsePostgres();
    o.UseBusOutbox();          // publishes only after the transaction commits
});
```

With this, `await publishEndpoint.Publish(evt)` inside your transaction is staged in the outbox table and relayed post-commit. Your existing EF Core + domain-events setup adopts this with minimal change.

## Exactly-once is a myth — say so confidently
Across system boundaries you get at-least-once. Duplicates *will* arrive (relay crash after publish, before mark-sent). The answer is **idempotent consumers**: a processed-message-ID table, natural idempotency (`UPDATE ... WHERE status = 'Pending'`), or upserts keyed on event ID. The phrase to use: *"effectively-once = at-least-once delivery + idempotent processing."*
