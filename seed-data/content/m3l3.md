# Sagas & Process Managers (MassTransit State Machines)

Sagas are the distributed answer to a question you've already solved locally: your ServiceOrder aggregate is a state machine *inside one service*. A saga is that same state machine **coordinating multiple services** — without distributed transactions.

## The concept
A long-running business process = sequence of local transactions across services. If step N fails, you don't roll back — you **compensate**: emit events that semantically undo earlier steps (cancel reservation, refund payment, release inventory).

- **Orchestration:** a central state machine sends commands and reacts to replies. Explicit, debuggable, one place to read the flow. (MassTransit's sweet spot.)
- **Choreography:** services react to each other's events, no coordinator. Loosely coupled but the flow exists nowhere — hard to reason about past ~3 steps. In interviews: *orchestration for complex flows, choreography for simple fan-out.*

## MassTransit state machine sketch
```csharp
public class OrderState : SagaStateMachineInstance
{
    public Guid CorrelationId { get; set; }
    public string CurrentState { get; set; } = default!;
    public Guid? PaymentId { get; set; }
}

public class OrderStateMachine : MassTransitStateMachine<OrderState>
{
    public State AwaitingPayment { get; private set; } = default!;
    public State Confirmed { get; private set; } = default!;

    public OrderStateMachine()
    {
        InstanceState(x => x.CurrentState);
        Initially(When(OrderSubmitted)
            .Then(ctx => ctx.Saga.CorrelationId = ctx.Message.OrderId)
            .PublishAsync(ctx => ctx.Init<ReservePayment>(new { ctx.Message.OrderId }))
            .TransitionTo(AwaitingPayment));
        During(AwaitingPayment,
            When(PaymentReserved).TransitionTo(Confirmed),
            When(PaymentFailed)
                .PublishAsync(ctx => ctx.Init<ReleaseInventory>(new { ctx.Saga.CorrelationId }))
                .Finalize());
    }
}
```
State persists to PostgreSQL via EF; correlation by OrderId routes events to the right instance. Add timeout handling with scheduled `OrderExpired` messages — mentioning timeouts unprompted is a senior tell.
