# LAB: Design a Ride-Hailing Backend (Careem-style)

A full mock you can rehearse aloud. Time-box 45 minutes, produce one diagram page + one notes page. Do it twice this week; the second run should feel mechanical.

## The prompt
*"Design the backend for a ride-hailing service in the Gulf: riders request trips, nearby drivers get matched, both see live location, riders pay in-app. 5M riders, 200K drivers, peak 2K ride-requests/min."*

## Required structure (practice this exact flow)
1. **Requirements:** functional (request, match, track, pay) + non-functional: matching < 30s, location updates ~4s fresh, payment exactly-once-effective, 99.95% availability.
2. **Capacity:** 2K req/min ≈ 35 RPS matching (small!) — but 200K drivers × location ping/4s = **50K writes/sec on location**. Say it: *location ingestion is the real scaling problem, not ride requests.*
3. **API + entities:** `POST /rides`, `GET /rides/{id}`, driver location ingest; Rider, Driver, Trip (state machine: Requested → Matched → InProgress → Completed — your ServiceOrder experience maps directly).
4. **High level:** mobile → gateway → Trip service (PostgreSQL), Location service (Redis geo / in-memory grid, *not* the relational DB), Matching service (geohash lookup of nearby drivers), Payments (queue + idempotency keys), Kafka for trip events feeding analytics/ETA.
5. **Deep dives to prepare:** geo-indexing (geohash cells vs quadtrees), surge/matching fairness, payment effectively-once, what fails when a region's Redis dies.

## Deliverable
Record yourself (audio is fine) running the full 45 minutes. Listen back once. You'll fix more in one listen than in three silent rehearsals.
