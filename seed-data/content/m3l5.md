# LAB: Event-Driven Order Pipeline (Portfolio Repo)

The module deliverable: a public GitHub repo you can open in interviews when messaging comes up. Target: 2–3 evenings.

## Build: "RelayFlow" — a miniature FastTrack-style booking pipeline

**Services (one solution, three deployables + docker-compose):**
1. **Booking.Api** (.NET Minimal API + PostgreSQL + EF outbox) — `POST /bookings` saves the booking and stages `BookingSubmitted` in the same transaction.
2. **Payments.Worker** — consumes `BookingSubmitted`, simulates a charge (random fail 20%), emits `PaymentSucceeded` / `PaymentFailed`. Idempotent via processed-event table.
3. **Booking saga** (hosted in Booking.Api) — state machine: Submitted → AwaitingPayment → Confirmed / Cancelled, with compensation (release capacity) on failure and a 2-minute timeout that auto-cancels.
4. **Kafka tap (stretch):** publish confirmed bookings to a Kafka topic; a tiny console "analytics" consumer counts confirmations per partner.

## Definition of done
- `docker compose up` brings up PostgreSQL, RabbitMQ, (Kafka), and all services.
- README with an architecture diagram (Mermaid is fine) and a **"failure tour"**: documented steps to kill the broker mid-flow, replay duplicates, and poison a message — each with the observed behavior.
- The failure tour is what makes this repo interview-gold: nobody else's portfolio shows *recovery*, only happy paths.

## Interview framing you've just earned
*"My production system uses in-process domain events; I built RelayFlow to take the same ServiceOrder state machine distributed — outbox for the dual-write problem, saga with compensation and timeouts, idempotent consumers because exactly-once is a myth."* That paragraph answers half of the messaging questions you'll ever get.
