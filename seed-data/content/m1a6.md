## 1. Heaps: partial order is cheaper than total order

The decision rule, stated as a trade: **sorting costs O(n log n) for total order; a heap costs O(n log k) for "just the k that matter."** When k ≪ n, that's the win. When you also need *streaming* answers (data keeps arriving), a heap is the only option among the basic tools.

### The top-k template — and the counterintuitive core
K largest elements → maintain a **min**-heap of size k. Yes, min-heap for largest: the root is the *weakest member of the elite*, so each newcomer fights exactly one gatekeeper:

```python
import heapq

def k_largest(nums: list[int], k: int) -> list[int]:
    heap = nums[:k]
    heapq.heapify(heap)                      # O(k)
    for n in nums[k:]:
        if n > heap[0]:                      # beats the weakest elite?
            heapq.heapreplace(heap, n)       # pop+push in one O(log k) op
    return heap
```

O(n log k) time, O(k) space — and it works on an infinite stream, which the sorting answer can't. Saying "heapify is O(n), not O(n log n)" when asked about building the initial heap is a precision point most candidates miss.

Python wrinkles to handle without blinking: `heapq` is min-only — negate values for max-heap behavior (`heappush(h, -x)`); tuples compare lexicographically, so `(priority, tiebreaker, payload)` — and you *need* the tiebreaker when payloads aren't comparable (the Merge-K-Lists crash from Lesson 4).

### Two named patterns to recognize
- **Two-heaps (Find Median from Data Stream):** a max-heap of the lower half and min-heap of the upper half, rebalanced to size-diff ≤ 1; median reads off the roots in O(1). This is a *streaming aggregate* — the same shape as a percentile tracker in a metrics pipeline, and you should say so.
- **Heap + lazy deletion:** when removing arbitrary elements, don't search the heap (O(n)); mark dead entries in a set and discard them when they surface at the root. Real schedulers do exactly this.

## 2. Backtracking: exhaustive search with a contract

Trigger words: *all combinations / permutations / subsets / paths*. The shape is always the same three-beat contract — **choose, explore, unchoose**:

```python
def subsets(nums: list[int]) -> list[list[int]]:
    res, path = [], []
    def backtrack(start: int):
        res.append(path[:])                  # snapshot! path keeps mutating
        for i in range(start, len(nums)):
            path.append(nums[i])             # choose
            backtrack(i + 1)                 # explore
            path.pop()                       # unchoose
    backtrack(0)
    return res
```

Three load-bearing details that decide whether your solution works:

1. **`path[:]` snapshot.** Appending `path` itself appends a *reference*; every entry in `res` ends up being the same (finally empty) list. The most common backtracking bug in existence.
2. **`start` index = "no looking back."** It's what makes these *combinations* (order-free) instead of permutations. For permutations you instead track a `used` set and iterate from 0 every time.
3. **Duplicate handling = sort + skip-same-level:** `if i > start and nums[i] == nums[i-1]: continue`. The condition means "skip a duplicate only if its twin was available at this same decision level" — be ready to explain that sentence, because it *will* be probed on Subsets II / Combination Sum II.

Complexity honesty: subsets is O(2ⁿ·n), permutations O(n!·n). Say it without apology — the *output itself* is exponential, so no algorithm does better. Then add the pruning sentence: "I'd prune branches early where the remaining sum can't reach the target" (Combination Sum) — pruning awareness is the senior layer on a junior pattern.

**Word Search** (grid + backtracking) merges this lesson with flood-fill: mark the cell in-place before exploring, restore on unchoose. The mark/restore *is* choose/unchoose — one pattern wearing the other's clothes.

## 3. Dynamic programming: a procedure, not an epiphany

DP has mythology around "seeing it." Replace the mythology with this five-step procedure, run in order, out loud:

1. **State** — what's the minimal description of a subproblem? (Usually: "best answer for first i items," or "for capacity c," or "for i with last choice j.")
2. **Recurrence** — answer for state i in terms of *smaller* states. This is the actual thinking step.
3. **Base cases** — smallest states answered directly.
4. **Order** — compute so dependencies are ready (usually left-to-right; or memoize and let recursion find the order).
5. **Space squeeze** — if state i only needs i−1 and i−2, two variables replace the array. Mention it even if you don't bother coding it.

### Worked: House Robber, narrated by the procedure
*State:* `dp[i]` = max loot from the first i houses. *Recurrence:* rob house i (`dp[i-2] + v[i]`) or skip it (`dp[i-1]`) — take the max. *Base:* `dp[0]=v[0]`, `dp[1]=max(v[0],v[1])`. *Order:* left to right. *Squeeze:*

```python
def rob(nums: list[int]) -> int:
    prev2 = prev1 = 0
    for n in nums:
        prev2, prev1 = prev1, max(prev1, prev2 + n)
    return prev1
```

Climbing Stairs is the same recurrence with + instead of max (and is literally Fibonacci); Min Cost Climbing Stairs adds weights. Three problems, one state design — that's the leverage of learning *states*, not problems.

### Worked: Coin Change — "unbounded choice" states
*State:* `dp[a]` = fewest coins to make amount a. *Recurrence:* `dp[a] = 1 + min(dp[a - c] for c in coins if c <= a)`. *Base:* `dp[0] = 0`. Infeasible states stay ∞; answer is `dp[amount]` or −1.

```python
def coin_change(coins: list[int], amount: int) -> int:
    dp = [0] + [float('inf')] * amount
    for a in range(1, amount + 1):
        for c in coins:
            if c <= a:
                dp[a] = min(dp[a], dp[a - c] + 1)
    return dp[amount] if dp[amount] != float('inf') else -1
```

Why greedy fails here — coins {1, 3, 4}, amount 6: greedy takes 4+1+1 (three coins), optimal is 3+3 (two). Have this counterexample loaded; "why not greedy?" is the standard probe, and answering with a concrete case in five seconds ends the line of questioning.

### The 2-D step: Longest Common Subsequence
*State:* `dp[i][j]` = LCS length of `a[:i]`, `b[:j]`. *Recurrence:* chars match → `1 + dp[i-1][j-1]`; else → `max(dp[i-1][j], dp[i][j-1])`. This single table is the engine behind diff tools, Git merge conflict detection, and edit distance — "this is how `git diff` works" is a sentence worth saying.

### When you can't see it (the part nobody teaches)
Live and stuck? Do this, in order: (1) solve brute-force recursion *first* — full marks for a correct exponential solution beats a broken DP; (2) then say "this recursion recomputes overlapping subproblems — I'll memoize," and add `@functools.cache` (top-down DP *is* DP, full credit at nearly every company); (3) only convert to bottom-up if asked or if recursion depth threatens. This de-risking sequence turns DP from a cliff into a ramp, and interviewers score the *process* visibly.

## 4. How this breaks in production

Top-k heaps run every "top 10 endpoints by latency" dashboard. Two-heaps is a streaming median/percentile estimator. Backtracking with pruning is constraint solving — seat assignment, configuration validation. DP is rate-shaping and resource allocation; LCS is your diff view in every PR review. None of this is interview-only knowledge; the interview just checks you can derive it on demand.
