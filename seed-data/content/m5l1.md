# The AZ-305 Mental Model: Requirements → Architecture

AZ-204 retired from your plan; AZ-305 (Designing Microsoft Azure Infrastructure Solutions) is the architect signal. But the certificate is secondary — the *mental model* is what interviews test: every answer maps **business requirements to service choices with stated trade-offs**.

## The requirement vocabulary
- **RPO** (Recovery Point Objective): max acceptable data loss, measured in time. "RPO 5 minutes" → replication, not nightly backups.
- **RTO** (Recovery Time Objective): max acceptable downtime. "RTO 1 hour" → warm standby; "RTO 5 min" → active-active.
- **SLA math:** chained dependencies multiply: 99.9% × 99.9% = 99.8%. Parallel redundancy improves: 1-(0.001²) ≈ 99.9999%. Do this multiplication aloud in interviews.
- **Compliance/residency:** drives region choice before anything else (UAE North for regulated Dubai workloads).

## The Azure decision ladders (know these cold)
- **Compute:** App Service (web default) → AKS (orchestration needs, your microservices) → Container Apps (serverless containers, KEDA built in) → Functions (event-driven bursts).
- **Data:** Azure SQL / PostgreSQL Flexible Server (relational core) → Cosmos DB (global, partition-key world) → Storage/Data Lake (blobs, analytics).
- **Resilience tiers:** single instance → zone-redundant (AZ failure) → multi-region paired with failover (region failure). Each tier roughly doubles cost — say the cost sentence; architects who never mention money don't get hired.
- **Messaging:** Service Bus (commands, sessions, DLQ) ↔ Event Hubs (Kafka-shaped streams) ↔ Event Grid (reactive pub/sub for Azure events). This maps 1:1 onto your M3 knowledge.
